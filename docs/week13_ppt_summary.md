# 13주차 PPT 작성용 결과 정리

## 4. 실험 구성 및 평가 방법

### 실험 목적
본 실험의 목적은 제조 업무 질의에서 일반 LLM, 단순 RAG, DB Tool 기반 Agent, 제안 RAG-Agent 구조의 차이를 비교하고, 문서 근거성과 정형 데이터 조회 정확도를 함께 평가하는 것이다.

### 실험 구성
| 구분 | 내용 |
| --- | --- |
| 대상 업무 | LOT 이력 조회, 생산 실적 조회, 불량률 확인, 설비 알람 조치 기준 질의 |
| 문서 데이터 | 작업표준서, 설비 알람 조치 표준서, LOT 추적 기준, 생산/품질 지표 산정 기준 |
| 정형 데이터 | 일자별 생산 실적, LOT 공정 이력, 설비 알람 및 조치 이력 |
| 비교 방식 | 일반 LLM, 문서 RAG, DB Tool Agent, Hybrid RAG-Agent |
| 실행 흐름 | 질문 입력 -> 질문 라우팅 -> RAG 검색/DB Tool 호출 -> 응답 생성 -> 근거 검증 |

### 평가 질의 유형
| 유형 | 예시 질문 | 평가 관점 |
| --- | --- | --- |
| 문서형 | ALM-101 과열 알람 조치 절차는? | 문서 검색 정확도, 근거 제시 |
| DB형 | 2026-06-03 LINE-A 생산 현황은? | 조회 조건 해석, 수치 정확도 |
| 복합형 | LOT-2026-A001 이력과 관련 알람 조치 기준은? | 문서와 DB 결과 통합 능력 |

### 평가 지표
| 평가 지표 | 설명 |
| --- | --- |
| 라우팅 정확도 | 질문을 문서형, DB형, 복합형으로 올바르게 분류했는지 평가 |
| 검색 적합도 | 질문과 관련 있는 작업표준서 또는 매뉴얼 청크를 상위 결과로 검색했는지 평가 |
| Tool 호출 정확도 | LOT ID, 일자, 라인, 설비 ID 등 조회 조건을 올바르게 추출했는지 평가 |
| 수치 정확도 | 생산 수량, 불량 수량, 불량률 등 DB 조회 결과가 원본 데이터와 일치하는지 평가 |
| 근거 제시율 | 답변에 문서 출처와 조회 결과가 함께 제시되었는지 평가 |
| 환각 억제 | 근거 없는 조치, 수치, 원인을 생성하지 않았는지 평가 |

### MVP 구현 환경
| 항목 | 내용 |
| --- | --- |
| 언어 | Python 3 |
| 데이터 저장소 | SQLite |
| RAG 검색 | 표준 라이브러리 기반 TF-IDF 검색 MVP |
| Agent 구성 | Rule-based Router + Safe Tool Registry + Answer Generator |
| 확장 방향 | Vector DB, Embedding API, MES/ERP/QMS API, 사내 LLM 연동 가능 |

## 5. 결과 및 분석

### 구현 결과 요약
본 프로젝트에서는 제조 업무 질의를 처리하기 위한 MVP RAG-Agent 구조를 구현하였다. 구현 결과, 질문 유형에 따라 문서 검색과 DB Tool 호출을 선택적으로 수행할 수 있었으며, 복합 질의에서는 두 결과를 함께 사용해 답변을 생성할 수 있었다.

| 구성 요소 | 구현 결과 |
| --- | --- |
| 질문 라우터 | 키워드와 식별자 기반으로 문서형, DB형, 복합형 질문 분류 |
| RAG 검색기 | 작업표준서, 알람 조치 문서, LOT 추적 기준, QMS 기준, 보전 매뉴얼에서 관련 문서 청크 검색 |
| DB Tool | LOT 이력, 일일 생산 실적, 품질 요약, 설비 알람 이력, ERP성 자재 재고 조회 |
| 응답 생성기 | 템플릿 응답, 로컬 Ollama 응답, 선택적 OpenAI 응답 중 선택 가능 |
| 확장성 | 실제 MES/ERP/QMS API로 Tool 계층 교체 가능 |

### 문제 정의와 코드 개선 반영
| PPT 문제 정의 | 코드상 개선 반영 | 설명 |
| --- | --- | --- |
| P1. 일반 LLM의 내부 데이터 부재·환각 위험 | `documents.py`, `vector_store.py` 기반 RAG 검색 | 표준서, 매뉴얼, QMS 기준을 검색하여 답변 근거로 제공 |
| P2. 단순 RAG의 정형 데이터 처리 한계 | `tools.py`의 사전 정의 DB Tool | LOT 이력, 생산 실적, 품질 요약, 설비 알람, 재고 현황 조회 가능 |
| P3. Text-to-SQL의 운영 안전성 문제 | `ManufacturingToolRegistry` 기반 Tool-Calling | 자연어를 임의 SQL로 변환하지 않고 승인된 조회 함수만 호출 |
| P4. 정형 데이터와 문서가 함께 필요한 복합 질의 | `router.py`와 `agent.py`의 hybrid 처리 | DB 조회 결과와 문서 근거를 하나의 응답으로 통합 |
| MES 중심에서 ERP/QMS 확장 필요 | `get_inventory_status`와 QMS/보전 문서 추가 | ERP성 재고 조회와 QMS 불량 기준 검색까지 샘플 확장 |
| 외부 API 의존성 최소화 필요 | `OllamaAnswerGenerator` 추가 | 로컬 Ollama 모델로 폐쇄망/개인 PC 환경에서도 LLM 응답 생성 가능 |

### 비교 분석
| 방식 | 장점 | 한계 |
| --- | --- | --- |
| 일반 LLM | 자연어 응답 생성이 쉬움 | 내부 생산 데이터와 최신 업무 기준을 알 수 없어 환각 위험이 큼 |
| 단순 RAG | 표준서와 매뉴얼 기반 설명에 강점 | 생산량, LOT 이력, 불량률 등 정형 데이터 조회가 어려움 |
| Text-to-SQL | 자연어 기반 DB 조회 가능 | 운영 DB 안전성 문제와 문서 근거 부족 문제가 발생할 수 있음 |
| 제안 RAG-Agent | 문서 근거와 정형 조회 결과를 함께 사용 | 라우팅 규칙과 Tool 정의 품질에 따라 성능이 좌우됨 |

### 주요 관찰 결과
- 문서형 질의에서는 알람 조치 표준서와 생산/품질 기준 문서를 검색하여 답변 근거를 제공할 수 있었다.
- DB형 질의에서는 일자, 라인, LOT ID 등 조회 조건을 추출하여 사전 정의된 Tool을 호출할 수 있었다.
- 복합형 질의에서는 LOT 공정 이력과 관련 알람 조치 기준을 함께 제시함으로써 단순 문서 QA보다 업무 활용성이 높았다.
- 임의 SQL 생성을 사용하지 않고 Tool Registry를 통해 조회 범위를 제한함으로써 운영 DB 안전성을 높일 수 있었다.
- 답변에 조회 조건과 문서 근거를 함께 표시하여 사용자가 결과의 출처를 확인할 수 있었다.

### 결과 해석
제안 구조는 제조 현장에서 자주 발생하는 “수치 조회 + 업무 기준 설명” 형태의 질문에 적합하다. 특히 LOT 이력, 설비 알람, 품질 기준처럼 정형 데이터와 비정형 문서가 함께 필요한 업무에서는 RAG와 Tool-Calling을 결합한 Hybrid Agent 방식이 단일 접근방식보다 효과적이다.

## 6. 한계점 및 토론

### 한계점
- 본 MVP는 실제 MES, ERP, QMS와 직접 연동하지 않고 샘플 SQLite 데이터베이스를 사용하였다.
- 샘플 데이터는 발표와 구조 검증을 위한 테스트 데이터이므로 실제 생산 현장의 데이터 품질, 지연, 누락, 권한 조건을 반영하지 못한다.
- 질문 라우팅은 규칙 기반이므로 복잡한 자연어 표현이나 현장 약어를 모두 처리하기 어렵다.
- TF-IDF 기반 검색은 외부 의존성이 없다는 장점이 있지만, 의미 기반 검색 성능은 임베딩 기반 Vector DB보다 제한적이다.
- 응답 생성기는 템플릿, 로컬 Ollama, OpenAI API 중 선택 가능하지만, 실제 성능은 선택한 모델 크기와 한국어/제조 도메인 이해도에 좌우된다.
- 권한 관리, 개인정보/영업비밀 보호, 감사 로그, 운영 장애 대응 등 실제 기업 적용에 필요한 보안 기능은 MVP 범위에서 제외하였다.

### 실 MES 적용 시 필요한 보완
| 보완 항목 | 필요 이유 |
| --- | --- |
| 실제 MES/QMS/ERP API 연동 | 샘플 SQLite 대신 운영 데이터 또는 Read Replica 기반 조회 필요 |
| 권한 기반 Tool 실행 | 사용자별 조회 가능 라인, 품목, 창고, LOT 범위 제한 필요 |
| 감사 로그 | 누가 어떤 조건으로 어떤 Tool을 호출했는지 추적 필요 |
| 데이터 마스킹 | 원가, 개인정보, 영업비밀 등 민감 정보 보호 필요 |
| Vector DB/GraphRAG | 설비, LOT, 공정, 불량 코드 간 관계 기반 검색 고도화 필요 |
| 운영 Guardrail | 조회 실패, 빈 결과, 오래된 데이터, 문서 버전 불일치 처리 필요 |

### 토론 및 개선 방향
- 실제 운영 적용을 위해서는 사내 데이터 권한 체계와 Tool 호출 권한을 연동해야 한다.
- 질문 라우터는 LLM 기반 Intent Classification 또는 Few-shot 분류로 확장할 수 있다.
- 문서 검색은 TF-IDF에서 Embedding 기반 Vector DB 또는 GraphRAG 구조로 발전시킬 수 있다.
- Tool은 단순 조회뿐 아니라 보고서 생성, 이상 LOT 후보 탐색, 알람 재발 분석 등 업무 단위 기능으로 확장할 수 있다.
- 응답 검증 단계에서 문서 출처, 조회 조건, 데이터 최신성, 사용자 권한을 함께 점검하는 Guardrail이 필요하다.

## 7. 결론 및 향후 연구

### 결론
본 프로젝트는 제조 기업의 분산된 문서 지식과 정형 업무 데이터를 자연어 인터페이스로 연결하기 위한 RAG-Agent MVP를 제안하고 구현하였다. 제안 구조는 질문 라우팅, RAG 검색, 안전한 DB Tool 호출, 근거 기반 응답 생성을 결합하여 일반 LLM의 환각 문제와 단순 RAG의 정형 데이터 처리 한계를 보완한다.

### 기대 효과
- 작업자는 여러 시스템 화면을 직접 이동하지 않고 자연어로 생산, 품질, 설비 정보를 확인할 수 있다.
- 답변에 문서 근거와 DB 조회 결과를 함께 제공하여 업무 신뢰성을 높일 수 있다.
- 사전 정의된 Tool 호출 구조를 사용함으로써 운영 DB에 대한 위험한 임의 질의를 줄일 수 있다.
- MES 중심 구조에서 ERP, QMS, 설비 모니터링 시스템으로 확장 가능한 Agent 기반 업무 인터페이스를 설계할 수 있다.

### 향후 연구
- 실제 제조 데이터와 문서를 이용한 정량 평가 데이터셋 구축
- Embedding Vector DB와 GraphRAG를 활용한 공정, 설비, LOT 관계 검색 고도화
- LLM 기반 질문 라우터와 Tool 선택 정책 개선
- 사용자 권한, 데이터 마스킹, 감사 로그를 포함한 운영 보안 구조 설계
- 현장 작업자 대상 사용성 평가와 기존 MES GUI 대비 업무 시간 단축 효과 분석
