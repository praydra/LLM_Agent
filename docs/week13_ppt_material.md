# 13주차 완성 및 토의 PPT 자료

본 문서는 기존 PPT의 4~7장 내용을 채우기 위한 발표용 자료이다. 슬라이드에 바로 옮길 수 있도록 제목, 핵심 메시지, 표/그림 구성, 발표 메모 형태로 정리했다.

---

## 4. 실험 구성 및 평가 방법

### Slide 4-1. 실험 목표와 비교 대상

**슬라이드 제목**
실험 목표: 제조 업무 질의에서 RAG-Agent 구조의 필요성을 검증

**핵심 메시지**
제조 현장 질의는 문서 지식과 정형 업무 데이터를 동시에 요구하므로, 일반 LLM 또는 단순 RAG만으로는 충분하지 않다.

**표 구성**
| 비교 대상 | 처리 방식 | 예상 한계 |
| --- | --- | --- |
| 일반 LLM | 모델 내부 지식으로 답변 생성 | 사내 최신 생산 데이터 부재, 환각 위험 |
| 단순 RAG | 표준서/매뉴얼 검색 후 답변 | LOT, 생산량, 불량률 등 DB 조회 한계 |
| DB Tool Agent | 사전 정의 Tool로 정형 데이터 조회 | 업무 기준, 조치 절차 등 문서 근거 부족 |
| 제안 RAG-Agent | 질문 라우팅 후 RAG + Tool 통합 | 라우터와 Tool 정의 품질에 의존 |

**발표 메모**
본 프로젝트는 단순 챗봇 구현이 아니라 제조 업무 질의의 특성을 반영한 구조 검증에 초점을 둔다. 특히 “LOT 이력은 DB에 있고, 알람 조치 기준은 문서에 있는” 상황을 주요 실험 시나리오로 설정했다.

---

### Slide 4-2. 샘플 데이터셋 구성

**슬라이드 제목**
샘플 데이터셋: 문서 지식 + 제조 업무 DB + 평가 질의 세트

**핵심 메시지**
MVP 검증을 위해 MES, QMS, 설비, ERP 성격의 샘플 데이터를 구성하고, 문서형/DB형/복합형 질의를 분리했다.

**표 구성**
| 데이터 구분 | 샘플 내용 | 코드 위치 |
| --- | --- | --- |
| 작업/알람 문서 | 알람 조치 표준서, LOT 추적 기준, 보전 매뉴얼 | `sample_data/knowledge_base` |
| QMS 문서 | 불량 코드, LOT 보류, 재작업 기준 | `qms_defect_handling_policy.md` |
| MES 데이터 | 일자/라인/품목별 생산량, 불량 수량 | `daily_production` |
| LOT 이력 | LOT별 공정, 설비, 검사 결과, 불량 코드 | `lot_history` |
| 설비 알람 | 알람 코드, 등급, 조치, 비가동 시간 | `equipment_alarm` |
| ERP성 재고 | 자재 코드, 창고, 가용/예약/할당 가능 수량 | `inventory_status` |
| 평가 질의 | 문서형, DB형, 복합형, 한계 확인형 | `sample_data/test_questions.json` |

**발표 메모**
실제 MES 연결 전 단계이므로 샘플 SQLite DB를 사용했다. 다만 테이블과 Tool은 실제 MES, QMS, ERP API로 교체 가능한 경계로 설계했다.

### Slide 4-3. 평가 질의 유형

**슬라이드 제목**
평가 질의: 현장 업무 질문을 4개 유형으로 구성

**표 구성**
| 유형 | 예시 질문 | 평가 관점 |
| --- | --- | --- |
| 문서형 | 재가동 기준과 작업자 기록 절차를 설명해줘 | RAG 검색 적합도, 근거 문서 제시 |
| DB형 | 2026-06-05 LINE-A 생산 현황과 불량률을 알려줘 | 조건 추출, Tool 호출, 수치 정확도 |
| 복합형 | LOT-2026-B015는 재작업이 있었는데 QMS 보류 기준상 어떻게 판단해야 해? | DB 결과와 문서 기준 통합 |
| 한계확인형 | 이 MVP는 실제 MES 운영 DB에 바로 연결되어 있나요? | 시스템 한계와 운영 적용 조건 설명 |

**평가 지표**
| 지표 | 설명 |
| --- | --- |
| 라우팅 정확도 | 질문을 document, data, hybrid로 분류했는지 |
| Tool 선택 정확도 | 필요한 사전 정의 Tool을 선택했는지 |
| 조건 추출 정확도 | LOT ID, 일자, 라인, 품목, 설비 ID, 알람 코드, 자재 코드 추출 여부 |
| 검색 근거성 | 관련 문서가 상위 검색 결과에 포함되는지 |
| 응답 근거성 | 조회 조건, 조회 결과, 문서 근거, 한계가 답변에 포함되는지 |
| 안전성 | 임의 SQL 없이 승인된 Tool만 호출하는지 |

## 5. 결과 및 분석

### Slide 5-1. 문제 정의와 코드 반영 결과

**슬라이드 제목**
PPT 문제 정의를 코드 구조에 반영한 결과

**핵심 메시지**
PPT에서 제시한 P1~P4 문제를 각각 RAG, Tool, Router, 통합 응답 생성 구조로 개선 반영했다.

**표 구성**
| PPT 문제 정의 | 코드상 반영 | 개선 효과 |
| --- | --- | --- |
| P1. 일반 LLM의 내부 데이터 부재와 환각 위험 | `TfidfVectorStore`, `load_documents`, 문서 근거 출력 | 답변에 표준서/매뉴얼 근거를 함께 제시 |
| P2. 단순 RAG의 정형 데이터 처리 한계 | `ManufacturingToolRegistry`, `get_lot_history`, `get_daily_production` | 생산량, LOT 이력, 불량률 등 수치 질의 처리 |
| P3. Text-to-SQL의 운영 DB 안전성 문제 | 임의 SQL 생성 금지, 사전 정의 Tool만 호출 | 위험 쿼리 가능성 감소, 조회 범위 명확화 |
| P4. 문서와 DB가 함께 필요한 복합 질의 | `QuestionRouter`의 `hybrid` 라우팅, Agent 통합 응답 | DB 조회 결과와 문서 기준을 함께 설명 |
| 확장성 요구 | `get_inventory_status` 추가 | MES 중심에서 ERP성 재고 조회로 확장 가능성 확인 |

**그림 아이디어**
문제 정의 P1~P4를 왼쪽에 배치하고, 오른쪽에 코드 구성요소를 연결하는 매핑 다이어그램.

```text
P1 환각 위험        -> RAG 검색 + 문서 근거
P2 정형 조회 한계   -> Safe DB Tool
P3 임의 SQL 위험    -> Tool Registry
P4 복합 질의        -> Router + Hybrid Agent
```

### Slide 5-2. 구현된 RAG-Agent 처리 흐름

**슬라이드 제목**
처리 흐름: 질문 라우팅 → RAG/Tool 실행 → 근거 기반 응답

**그림 구성**
```text
사용자 질문
  ↓
QuestionRouter
  ├─ document: 지식문서 검색
  ├─ data: DB Tool 호출
  └─ hybrid: 문서 검색 + DB Tool 호출
  ↓
ManufacturingRagAgent
  ↓
AnswerGenerator
  ↓
근거 문서 + 조회 결과 + 한계 포함 응답
```

**코드 근거**
| 구성 요소 | 파일 | 역할 |
| --- | --- | --- |
| 질문 라우터 | `manufacturing_rag_agent/router.py` | 질문 유형 분류와 Tool 후보 생성 |
| 문서 검색 | `documents.py`, `vector_store.py` | 문서 로딩, 청킹, TF-IDF 검색 |
| Tool 호출 | `tools.py` | MES/QMS/설비/ERP성 데이터 조회 |
| Agent 조합 | `agent.py` | 검색 결과와 Tool 결과를 통합 |
| 응답 생성 | `llm.py` | 템플릿, 로컬 Ollama, OpenAI 기반 응답 생성 |

**발표 메모**
현재는 외부 의존성을 줄이기 위해 TF-IDF와 템플릿 응답을 기본값으로 사용한다. 다만 `OllamaAnswerGenerator`를 추가하여 로컬 Ollama 모델로 답변을 생성할 수 있고, 같은 인터페이스를 통해 OpenAI 또는 사내 LLM으로도 교체 가능하게 했다.

### Slide 5-3. 대표 실행 결과

**슬라이드 제목**
대표 결과: 복합 질의에서 LOT 이력과 알람 조치 기준을 통합

**예시 질문**
`LOT-2026-A001 이력과 관련 알람 조치 기준을 알려줘`

**실행 결과 요약**
| 항목 | 결과 |
| --- | --- |
| 라우팅 | `hybrid` |
| 호출 Tool | `get_lot_history`, `get_equipment_alarms` |
| DB 조회 결과 | LOT 공정 이력 3건, 관련 설비 알람 2건 |
| 검색 문서 | 설비 알람 조치 표준서, LOT 추적 업무 표준 |
| 응답 특성 | 공정 이력, 검사 결과, 알람 이력, 조치 기준을 함께 제시 |

**분석**
단순 RAG는 LOT 공정 이력을 알 수 없고, DB Tool만 사용하면 알람 조치 기준을 설명하기 어렵다. Hybrid RAG-Agent는 두 정보를 함께 사용하여 현장 사용자가 판단할 수 있는 근거를 제공한다.

## 6. 한계점 및 토론

### Slide 6-1. 현재 MVP의 한계

**슬라이드 제목**
한계점: 샘플 환경 검증이며 운영 적용 기능은 추가 필요

**표 구성**
| 한계 | 현재 상태 | 실제 적용 시 보완 |
| --- | --- | --- |
| 실제 MES 미연동 | SQLite 샘플 DB 사용 | MES/ERP/QMS API 또는 Read Replica DB 연결 |
| 데이터 규모 제한 | 테스트용 소량 데이터 | 실제 생산/품질/설비 이력 기반 평가셋 구축 |
| 라우팅 방식 | 키워드/정규식 기반 | LLM 기반 Intent Classification 또는 학습형 라우터 |
| 검색 방식 | TF-IDF MVP 검색 | Embedding Vector DB, GraphRAG 적용 |
| 응답 생성 | 템플릿 기본값, 로컬 Ollama 선택 가능 | 모델 품질 검증, 사내 LLM 또는 OpenAI 모델 연동 |
| 보안 | 권한/감사 로그 미구현 | 사용자 권한, 데이터 마스킹, 감사 로그, 승인 Workflow |
| 운영 안정성 | Tool 호출 실패 처리 단순 | 재시도, 장애 격리, 조회 Timeout, Observability |

**발표 메모**
현재 구현은 운영 시스템이 아니라 “구조 검증용 MVP”이다. 따라서 결과가 실제 MES 정확도를 의미하지 않으며, 실 MES 적용 전에는 보안/권한/데이터 품질 검증이 필요하다.

---

### Slide 6-2. 토론: 왜 Text-to-SQL 대신 Safe Tool인가

**핵심 메시지**
제조 운영 DB에서는 편의성보다 안전성과 감사 가능성이 중요하므로, MVP에서는 임의 SQL 생성 대신 승인된 Tool 호출 구조를 선택했다.

**비교 표**
| 방식 | 장점 | 위험 |
| --- | --- | --- |
| Text-to-SQL | 유연한 질의 가능 | 잘못된 SQL, 과도한 조회, 권한 우회 가능성 |
| Safe Tool | 조회 범위와 조건 통제 가능 | 사전에 Tool을 설계해야 함 |

**토론 포인트**
- 현장 적용에서는 “무엇이든 물어볼 수 있음”보다 “허용된 업무 범위에서 정확히 조회함”이 중요하다.
- Tool 호출 로그를 남기면 감사와 장애 분석이 가능하다.
- 향후에는 Tool 후보를 LLM이 선택하되, 실행은 승인된 Tool Registry를 통해 제한하는 구조가 적합하다.

## 7. 결론 및 향후 연구

### Slide 7-1. 결론

**슬라이드 제목**
결론: 제조 업무에는 RAG와 Agent Tool-Calling의 결합이 필요

**핵심 문장**
본 프로젝트는 제조 기업의 분산된 문서 지식과 정형 업무 데이터를 자연어 인터페이스로 연결하기 위한 RAG-Agent MVP를 구현하였다.

**주요 성과**
- 질문을 문서형, DB형, 복합형으로 분류하는 라우터 구현
- 작업표준서, 알람 조치 문서, QMS 기준을 검색하는 RAG 구조 구현
- LOT 이력, 생산 실적, 품질 요약, 설비 알람, ERP성 재고 조회 Tool 구현
- 임의 SQL 대신 사전 정의 Tool을 호출하여 운영 DB 안전성 관점을 반영
- 복합 질의에서 문서 근거와 조회 결과를 함께 제시하는 응답 구조 구현

---

### Slide 7-2. 향후 연구 및 적용 로드맵

**슬라이드 제목**
향후 연구: 샘플 MVP에서 실 제조 시스템 연계 Agent로 확장

**로드맵**
| 단계 | 내용 | 기대 결과 |
| --- | --- | --- |
| 1단계 | 실제 MES/QMS/ERP Read API 연결 | 샘플 DB 한계 해소 |
| 2단계 | 문서 임베딩 Vector DB 구축 | 의미 기반 검색 정확도 향상 |
| 3단계 | LLM 기반 라우터와 Tool 선택 정책 적용 | 복잡한 현장 표현 처리 |
| 4단계 | 권한, 마스킹, 감사 로그 적용 | 운영 보안 요구사항 충족 |
| 5단계 | 현장 사용자 평가 | 기존 MES GUI 대비 업무 시간 단축 검증 |

**마무리 메시지**
이번 MVP는 완성형 운영 시스템은 아니지만, 제조 현장 LLM 적용에서 필요한 핵심 구조인 “근거 검색, 안전한 데이터 조회, 복합 질의 처리”를 코드로 검증했다는 점에 의미가 있다.
